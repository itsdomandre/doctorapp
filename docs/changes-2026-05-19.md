# Mudanças — 19 de Maio de 2026

Sessão de revisão e correção de bugs no backend e implementação de páginas no frontend.

---

## Backend (`doctorapp`)

### Correções de segurança e bugs

**Circular dependency — SecurityConfig / JwtAuthenticationFilter / UserService**
- `PasswordEncoder` e `AuthenticationManager` movidos para `AppConfig`, quebrando o ciclo de injeção.

**Reutilização de tokens de ativação e reset**
- `AuthService.activateAccount` e `resetPassword` agora verificam a blacklist (`InvalidToken`) antes de processar, impedindo que um link já utilizado funcione uma segunda vez.
- O commit anterior só guardava os tokens na blacklist mas não verificava.

**Tokens de ativação/reset usáveis como tokens de acesso**
- `JwtAuthenticationFilter` passou a rejeitar tokens com claim `type` (ativação/reset), que só devem servir para os seus respetivos endpoints.

**`/me` endpoint — `isAuthenticated` sem parênteses**
- `@PreAuthorize("isAuthenticated")` corrigido para `@PreAuthorize("isAuthenticated()")`.

**`AuthController.resetPassword` — strings diretas no controller**
- Substituídas por `InvalidTokenException` (header ausente ou inválido) e `PasswordMismatchException` (senhas não coincidem), seguindo a convenção do projeto.
- `GlobalExceptionHandler` atualizado: `InvalidTokenException` 409 → 400, handler de `PasswordMismatchException` adicionado (400).

### API de agendamentos

**`POST /api/appointment/create` → `POST /api/appointment`**
- Alinhado com a convenção REST.

**`GET /api/appointment/my-appointments` com paginação e filtro de status**
- Antes devolvia `List<AppointmentDTO>` sem parâmetros.
- Agora devolve `Page<AppointmentDTO>` e aceita `?page`, `?size`, `?status`.
- Dois novos métodos no `AppointmentRepository`: `findByPatient(pageable)` e `findByPatientAndStatus(pageable)`.

### Testes

- `AuthenticationApiTest` reescrito: campo `login` → `email`, helper `loginPayload` parametrizado, teste quebrado substituído por três testes corretos (`validActiveCredentials`, `unverifiedAccount`, `wrongPassword`).

### Docker

- Credenciais do PostgreSQL em `docker-compose.yml` passaram de variáveis de ambiente para valores fixos de desenvolvimento.

## Frontend (`doctorapp-fe`)

### Correções de alinhamento com o backend

**`ResetPassword.tsx`**
- Verificação de `status === 409` removida; mensagem de erro unificada alinhada com o novo 400 do backend.

**`Sidebar.tsx`**
- Iniciais e nome de exibição passaram a usar `fullName` em vez do email.

### Appointments — alinhamento completo com o backend

Os contratos de API estavam completamente dessincronizados. Corrigido:

| Ponto | Antes | Depois |
|---|---|---|
| URL create | `POST /api/appointment` | Correto após fix do backend |
| Payload create | `{ date, time, specialty, reason }` | `{ dateTime, procedure, notes }` |
| GET my-appointments | Esperava `Page<T>`, backend devolvia `List` | Ambos agora `Page<T>` |
| Cancel | `POST` | `PUT` |
| Status enum | `PENDING, APPROVED, REJECTED, CANCELLED` | `REQUESTED, APPROVED, REJECTED, CANCELLED, COMPLETED` |
| Campos do DTO | `specialty, reason, date, time` | `procedure, dateTime, notes, patientName, doctorName` |

**`types/appointment.ts`** — tipos reescritos: `AppointmentStatus`, `Procedure` (enum com 11 valores), `PROCEDURE_LABELS` (labels em PT-BR), `Appointment`.

**`lib/api/appointments.ts`** — URLs, métodos e tipos corrigidos; `fetchAvailableSlots`, `fetchAllAppointments`, `fetchPendingAppointments`, `fetchTodayAppointments`, `searchAppointments`, `updateAppointmentStatus` adicionados.

**`StatusBadge.tsx`** — suporte a todos os 5 status com labels em PT-BR.

**`NewAppointment.tsx`** — formulário reescrito:
- Seleção de data → busca slots disponíveis do backend → select de horário
- Select de procedimento com os 11 valores do enum `Procedures`
- Campo de notas opcional

**`MyAppointments.tsx`** — filtros e botão de cancelar corrigidos para usar `REQUESTED` em vez de `PENDING`.
